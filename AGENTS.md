# 项目协作规范

## 工作流程

- 每次完成一个功能改动后，先运行与改动范围匹配的验证命令。
- 验证通过后，在本轮工作末尾创建 git commit。
- 不要把 IDE 临时文件、构建产物、密钥或本地环境文件加入提交。
- 如果验证无法运行，需要在回复里说明原因，并仍然提交已确认的配置或代码改动，除非用户明确要求暂不提交。

## Commit Message 规范

提交信息必须使用中文，并带 type 前缀：

```text
type: 中文描述
```

常用 type：

- `feat`: 新功能或新能力
- `fix`: Bug 修复
- `refactor`: 重构，不改变行为
- `docs`: 文档
- `test`: 测试
- `build`: 构建、依赖、脚手架
- `chore`: 杂项维护

示例：

```text
build: 初始化 Spring Boot RAG 脚手架
docs: 补充本地启动说明
feat: 增加文档解析入口
fix: 修复 Redis 配置读取问题
```
